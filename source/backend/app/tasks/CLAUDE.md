# Tasks (backend)

Owns one object type, `task.py` — the single shape every task uses, whatever its origin:
title, description, project, `workspaceId` (optional, empty means unassigned — a cross-cutting
collection tag, unrelated to `projectId`; see `app/workspaces/CLAUDE.md`), optional `blockId`
(empty means one-off; non-empty means this Task is nested inside that `Block`, see
`app/blocks/CLAUDE.md`), `parentTaskId` (empty means top-level; non-empty means this Task is a
subtask belonging to another Task — nesting is capped at a single level, enforced in
`task_routes.py`, not `task.py`), tags, `dueDate`, `scheduledAt`, `isComplete`/`completedAt`,
`isSkipped`/`isSkippable`, `order`, `durationBucket`, and `estimatedMinutes`. `task.py` is the
reference shape every other object module copies — read it before adding a new object type
anywhere in the app.

`order` is meaningful for an ad-hoc top-level task (no project, not block-nested, not a subtask)
and, separately, for a subtask (scoped to its siblings sharing the same `parentTaskId`) and for a
block-nested task (scoped to its siblings sharing the same `blockId`) — `task.create` computes it
server-side in all three cases, appending to the end of the relevant sibling set; a project task
still gets `order = 0` and nothing ever sorts by it there. It's the drag-and-drop reorder field
for the Tasks page's Ad-hoc sidebar group and for a task's Subtasks checklist. `order` is never
client-supplied on create, only via `PATCH .../order`. `durationBucket` is a rough bucketed time
estimate (`under5min`..`over60min`), editable from `TaskDetailPane`'s Settings card, distinct from
`estimatedMinutes` — a precise minutes value populated only for tasks generated from a block's
`BlockTaskTemplate` (see `app/blocks/CLAUDE.md`), used to compute a block's scheduled-vs-allotted
time on the Today page.

A block-nested task is generated by a `BlockTaskTemplate` belonging to a `BlockTemplate`
(`app/blocks/block_task_template.py`/`block_template.py`) — see `app/journals/today_engine.py` for
"which block templates occur today, and which of their nested task templates are due today, stamp
out a fresh Task." Once generated, a block-nested task is an ordinary `Task` row,
indistinguishable in shape from a one-off one; only `blockId` marks where it came from. Missed
occurrences are not carried forward — a missed day's tasks just become ordinary task history, see
`today_engine.py`.

`task_routes.py` exposes standard CRUD for `Task`, including `isComplete`/`isSkipped`, plus the
rule that setting one of those two to `true` clears the other. `?parentTaskId=` on
`GET /api/tasks` lists one task's direct subtasks, and `?blockId=` lists one block's nested tasks,
both mirroring the existing `?projectId=` filter.

Deleting a task (`DELETE /api/tasks/<id>`) cascades to delete all of its subtasks (a single
non-recursive pass, since nesting is capped at one level) — subtasks have no meaning without
their parent task. A task's `isComplete` can't be set to `true` (see `task_routes.update_task`)
while any of its subtasks are neither complete nor skipped. `isSkippable` governs whether a task
can be skipped at all — it defaults to `true` for a block-nested task or a subtask (unless a
`BlockTaskTemplate` overrides it to `false` for a keystone habit), `false` for a plain ad-hoc
top-level task (see `task.create`), and is user-editable afterward except for subtasks, which are
always skippable. A block's checklist is defined once on its `BlockTaskTemplate` set and copied
onto each freshly generated `Block` instance's own independent nested tasks — editing one
instance's tasks never affects the template or any other instance.
