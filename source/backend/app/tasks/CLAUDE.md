# Tasks (backend)

Owns two object types: `task.py` (the single shape every task uses, recurring-generated or not —
title, description, project, optional `recurring_template_id` (empty means one-off), tags,
`dueDate`, `scheduledAt`, and `isComplete`/`completedAt`) and `subtask.py` (a checklist item
belonging to one task). `task.py` is the reference shape every other object module copies — read
it before adding a new object type anywhere in the app.

A recurring task is generated by a `RecurringTaskTemplate`
(`app/recurrence/recurring_task_template.py`) — see `app/journals/today_engine.py` for "which
templates apply today, stamp out a fresh Task instance." Once generated, a recurring instance is
an ordinary `Task` row, indistinguishable in shape from a one-off one; only `recurringTemplateId`
marks where it came from. Recurring misses are not carried forward — a missed day's instance just
becomes ordinary task history, see `today_engine.py`.

`task_routes.py` exposes standard CRUD for `Task`, including `isComplete`. `subtask_routes.py`
exposes standard CRUD for `Subtask`, plus the rule that setting `isComplete`/`isSkipped` clears
the other one.

Deleting a task (`DELETE /api/tasks/<id>`) cascades to delete all of its subtasks via
`subtask.delete_all_for_task` — subtasks have no meaning without their parent task. A task's
`isComplete` can't be set to `true` (see `task_routes.update_task`) while any of its subtasks are
neither complete nor skipped. A recurring task's checklist is defined once on its
`RecurringTaskTemplate` (`subtask_titles`) and copied onto each freshly generated instance's own
independent Subtasks — editing one instance's subtasks never affects the template or any other
instance.
